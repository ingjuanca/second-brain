> Avancemos ahora hacia temas avanzados en torno a la programación _front-end_ frente a la de _back-end_. Más adelante en el curso cubriremos cómo generar código realmente con Next.js aquí. Y lo que es Next.js es esencialmente un marco de trabajo (_framework_) que está construido sobre React. Es decir, está construido sobre la biblioteca React.
> 
> Y Next.js proporciona estructura y características adicionales que React por sí solo no ofrece. Con esto, Next.js de hecho desdibuja la línea entre el _front-end_ y el _back-end_. Permite el desarrollo _full-stack_ dentro de un solo marco de trabajo, de modo que no necesitas separar tu código de _front-end_ y de _back-end_. Todo se maneja esencialmente dentro de Next.js.
> 
> Así que con Next.js se ofrece renderizado del lado del servidor (conocido como SSR) así como generación de sitios estáticos (conocido como SSG). Y estas son habitualmente funcionalidades de _back-end_. Next.js también incluye una característica de rutas de API, la cual te permite construir tus puntos finales (_endpoints_) de API como parte de tu aplicación de Next.js. Next.js también funciona bien con arquitecturas sin servidor (_serverless_), por lo que te permiten desplegar funcionalidad de _back-end_ sin gestionar un servidor tradicional. Sé que todo eso podría sonar como una ensalada de palabras.
> 
> ¿And qué significa exactamente todo eso? Solo significa que puedes encargarte esencialmente de muchas tareas de _back-end_ sin usar un marco de trabajo de _back-end_ separado, o un _back-end_ como servicio. De modo que al usar Next.js, puedes hacer esencialmente todo lo que Firebase u otro servicio de _back-end_ hace por ti. Lo único con lo que necesitarías combinar Next.js es con una base de datos real. Así que lo único que Next.js no puede manejar por sí solo es tener una base de datos separada para almacenar información. Pero aparte de eso, puedes pensar en Next.js como el simple hecho de combinar tanto la cocina del _back-end_ como el diseño del _front-end_ en un solo marco de trabajo.
> 
> Si avanzamos por aquí, podemos ver que Next.js proporciona las herramientas para interactuar con bases de datos pero en realidad no las reemplaza. Uh, Next.js no proporciona almacenamiento de datos persistente. Esto significa que cualquier almacenamiento de datos dentro de tu aplicación Next.js es solo temporal y usualmente solo dura el tiempo de la sesión. De modo que si estás manejando cosas como información de usuarios o datos, simplemente tienes que conectar una base de datos externa para guardar realmente esta información. Estaremos mostrando ejemplos de todo esto. Pero típicamente una de estas bases de datos externas suele ser simplemente Postgres. Y entonces con esto tendrás esencialmente el _front-end_ y el _back-end_ combinados en un solo marco de trabajo.
> 
> En pocas palabras, Next.js es un marco de trabajo moderno que te permite programar proyectos _full-stack_. Es decir, _front-end_ y _back-end_ dentro de un solo marco de trabajo. Y este tiene características aún más avanzadas para construir una aplicación web _full-stack_. Esto no significa, sin embargo, que no puedas conectar un servicio de _back-end_ dedicado. Por ejemplo, si fueras a construir una aplicación con una lógica de negocio compleja o requisitos de alto rendimiento, podrías querer aún un servicio de _back-end_ dedicado como Firebase, y de hecho puedes conectar un servicio de _back-end_ dedicado a tu aplicación Next.js. Y como mencioné, puedes incluso usar un servicio de _back-end_ basado en la nube como Firebase con Next.js para obtener funcionalidad adicional.
> 
> Ahora vayamos adelante y unamos todo para mostrarte un marco de trabajo típico de lo que estaremos construyendo. Cuando programemos con Claude, un marco de trabajo típico (que de nuevo es solo una tecnología que usaremos) se verá así para el _front-end_ : construiremos todo dentro de React, al menos al inicio. Y recuerden de nuevo que React es solo la biblioteca que trabaja con HTML, CSS y JavaScript. Combina los tres pilares tradicionales de la programación _front-end_ en una sola biblioteca. Uh, para nuestro _back-end_, usaremos la forma más simple de un _back-end_, que de nuevo es Firebase, el cual esencialmente manejará todo por nosotros. Um, es un _back-end_ como servicio basado en un servicio en la nube. Así que en realidad no necesitamos programar nada aquí. Solo tenemos que configurar una cuenta de Firebase para manejar todo, desde los servidores hasta la autenticación de usuarios así como el almacenamiento.
> 
> Y luego estaremos construyendo realmente este código de _front-end_ y _back-end_, todo dentro de un IDE. Y para empezar usaremos Replit, porque de nuevo, Replit es el IDE más simple dado que es basado en la nube, por lo que no necesitamos guardar realmente nada directamente en nuestra laptop o computadora con la que estemos trabajando. Todo estará simplemente configurado en la nube, y tomará menos pasos de configurar cuando se compara con algo como Cursor o VS Code. Pero de nuevo, actualizaremos nuestras configuraciones a medida que avancemos a lo largo del programa.
> 
> Una vez que tengamos todo esto esencialmente dominado, podemos conectar el _front-end_ y el _back-end_ construyendo una app con Next.js. Y luego para el IDE, pasaré a mostrarte cómo manejar realmente un entorno local para que tengas todas las herramientas al alcance de tus dedos, de modo que sepas exactamente cómo construir con Claude o cualquier otro LLM, sin importar qué tipo de aplicación quieras construir, o sin importar qué tipo de proceso quieras seguir al construir realmente estas aplicaciones.

## 💡 Ideas más relevantes

- **El potencial de Next.js:** Es un marco de trabajo (_framework_) avanzado construido sobre la biblioteca React. Su principal ventaja es que unifica el desarrollo _front-end_ y _back-end_ en un solo lugar, permitiendo la creación de proyectos _full-stack_ sin tener que separar los entornos de código.
    
- **Capacidades de servidor integradas:** Next.js ejecuta tareas tradicionales de _back-end_ dentro de su propia estructura, tales como el renderizado del lado del servidor (SSR), la generación de sitios estáticos (SSG), la creación de puntos finales de API (_API routes_) y soporte para arquitecturas sin servidor (_serverless_).
    
- **La necesidad de una base de datos externa:** A pesar de su robustez, Next.js no cuenta con almacenamiento persistente de datos por sí mismo (la información guardada es temporal y dura solo la sesión). Por lo tanto, requiere conectarse a una base de datos externa de forma obligatoria (como Postgres) para guardar datos a largo plazo.
    
- **Flexibilidad e integración híbrida:** Aunque Next.js puede sustituir las tareas de un _back-end_ básico , es totalmente compatible con servicios dedicados en la nube como Firebase. Esto resulta ideal si la aplicación exige un rendimiento muy alto o lógicas de negocio sumamente complejas.
    
- **Estructura y evolución de la enseñanza:**
    
    - **Fase inicial:** Los estudiantes comenzarán creando interfaces en **React** (que une HTML, CSS y JavaScript) junto con un _back-end_ preconfigurado en la nube vía **Firebase** para no tener que programar el servidor desde cero. Todo esto se trabajará en **Replit** debido a su simplicidad al estar en la nube.
        
    - **Fase avanzada:** Una vez dominados los conceptos iniciales, se dará el salto hacia **Next.js** para unificar el _full-stack_ y se migrará a un **entorno de desarrollo local** (como Cursor o VS Code) para enseñar al alumno a construir proyectos de nivel profesional guiados por IA.
